BUG #001
DATE FOUND: 7/2/2026
FOUND BY: Claude (during the code audit Nimith requested)
STATUS: FIXED
-------------------------------------------
WHAT WAS WRONG:
- The 20-artwork cap trigger (enforce_artwork_limit) could be beaten by
  two inserts running at the same time. Both would count 19 existing
  artworks, both would pass the check, and the artist would end up with
  21 artworks.

CAUSE (root cause, if known):
- Classic check-then-act race: the trigger counted rows and then allowed
  the insert, but nothing stopped a second transaction from doing the
  same count in parallel before either insert was committed. The count
  can't see the other transaction's uncommitted row.

HOW IT WAS FIXED:
- The trigger now locks the artist's row in tbl_artists (select ... for
  update) before counting. Two inserts for the same artist are forced to
  run the check one at a time, so the second one sees the first one's
  row and gets rejected. The function was made security definer so the
  lock works regardless of the caller's RLS policies.
- Verified with a live two-session test on local Postgres 16: artist at
  19 artworks, two concurrent inserts, second one correctly blocked then
  rejected, final count exactly 20.
- The new report flood cap trigger (enforce_report_limit) uses the same
  lock-then-count pattern so it does not inherit the same bug.

FILES CHANGED:
- supabase/migrations/20260701000002_create_tbl_artworks.sql
- supabase/migrations/20260701000003_create_tbl_reports.sql (same
  pattern applied preventively)

FIXED BY: Claude (approved by Nimith) on 7/2/2026

BUG #002
DATE FOUND: 7/2/2026
FOUND BY: Nimith (saw "server error" on the fresh Vercel deploy)
STATUS: FIXED
-------------------------------------------
WHAT WAS WRONG:
- The deployed site returned a server error on every page when the
  Supabase environment variables were not set yet (fresh Vercel deploy
  before configuration).

CAUSE (root cause, if known):
- src/proxy.ts (session middleware, runs on every request) passed the
  env vars to createServerClient with a non-null assertion. With the
  vars missing, the client constructor threw, which killed every request
  with a 500 before any page could render. Pages had graceful fallbacks,
  but the proxy ran first and never let them load.

HOW IT WAS FIXED:
- proxy.ts now checks the two env vars first; if missing, it logs the
  problem server-side and skips session handling, so pages render in
  their logged-out state with friendly "could not load" messages.
- Verified by running the production build with zero env vars: homepage
  and login return 200 and show the fallback text instead of a 500.
- Note: this makes a misconfigured deploy degrade gracefully. The real
  deploy still needs the env vars set in Vercel to actually work.

FILES CHANGED:
- src/proxy.ts

FIXED BY: Claude on 7/2/2026
